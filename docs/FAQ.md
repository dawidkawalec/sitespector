## 🤔 General Questions

### Q: Dlaczego Railway, a nie AWS/Azure/GCP?

**A:** Railway to najlepszy wybór dla MVP z kilku powodów:
- **Prostota:** Zero-config deployment (git push = deploy)
- **Koszt:** $30-50/miesiąc vs $100+ na AWS
- **Managed Database:** PostgreSQL w cenie, zero ops
- **Auto-scaling:** Automatyczne skalowanie przy wzroście ruchu
- **DX:** Najlepsza dev experience na rynku

Dla skali 1000 audytów/miesiąc Railway jest wystarczający. Przy 10k+ audytów można rozważyć migrację na AWS z Kubernetes.

---

### Q: Dlaczego FastAPI, a nie Django?

**A:** FastAPI jest lepsze dla tego projektu:
- **API-first:** Zaprojektowane dla REST API (Django to full framework)
- **Async:** Native async/await (ważne dla długich audytów)
- **Performance:** 2-3x szybszy niż Django
- **Type Safety:** Pydantic models = automatyczna walidacja
- **Auto Docs:** Swagger UI out-of-the-box

Django byłby overkill - nie potrzebujemy admin panel, ORM jest za ciężki.

---

### Q: Jak długo trwa jeden audyt?

**A:** Średnio **5-10 minut**:
- Screaming Frog crawl: 2-3 min
- Lighthouse (desktop + mobile): 1-2 min
- AI analysis (Claude): 1-2 min
- PDF generation: 30s - 1 min
- Competitor analysis (x3): +3-5 min

Dla prostej strony (10-20 stron) może być szybciej (3-5 min).

---

### Q: Ile kosztuje Claude API per audyt?

**A:** Około **$0.12 per audyt** (MVP):
- Input tokens: ~15k × $3/1M = $0.045
- Output tokens: ~5k × $15/1M = $0.075
- **Total:** ~$0.12

Przy 1000 audytów/miesiąc = $120/miesiąc na AI.
Przy przychodach 29 PLN/audit (~$7 USD) × 1000 = $7000, margin = 98%.

## 💻 Technical Questions

### Q: Jak obsługujemy strony wymagające JavaScript (SPA)?

**A:** 
- **Screaming Frog CLI:** Ma wbudowany renderer JS (Chrome headless)
- **Lighthouse:** Również używa Chrome headless
- **Fallback:** Jeśli SF nie działa, możemy użyć Playwright

99% stron zadziała out-of-the-box.

---

### Q: Co z stronami za loginowaniem (paywall)?

**A:** MVP **nie obsługuje** stron wymagających logowania:
- User dostanie error: "Cannot access protected page"
- Feature request na Etap 2: "Auth credentials input"

---

### Q: Jak skalujemy przy 10k audytów/dzień?

**A:** Plan skalowania:
1. **Do 100 audytów/dzień:** Jeden worker wystarczy
2. **100-1000/dzień:** Dodaj Redis queue + 5 workers
3. **1000-10k/dzień:** Kubernetes + horizontal scaling (10-50 workers)
4. **10k+/dzień:** Distributed architecture (multiple regions)

Railway pozwala łatwo dodawać workers (każdy = nowy service).

---

### Q: Czy dane audytów są cached?

**A:** MVP: **NIE** (każdy audyt jest fresh).

Etap 2 może dodać:
- Cache URL przez 24h (jeśli user zażąda tego samego URL)
- "Re-run audit" button (forced refresh)

Dla większości use cases fresh data > cached data.

---

### Q: Co jeśli Claude API spadnie?

**A:** Error handling strategy:
1. **Retry logic:** 3 próby z exponential backoff
2. **Graceful degradation:** Jeśli AI fail, zwróć tylko technical data (bez recommendations)
3. **Status:** Audit status = "completed_partial" (user widzi co się udało)
4. **Manual retry:** User może kliknąć "Re-analyze with AI"

Uptime Claude API: ~99.9%, więc rzadko problem.

## 🔐 Security Questions

### Q: Czy przechowujemy scraped HTML?

**A:** MVP: **Nie** (tylko wyparsowane metadane).
- Oszczędność storage
- Privacy-friendly
- GDPR compliant

Etap 2: Optional "Store raw HTML" feature (dla re-analysis).

---

### Q: Co z GDPR compliance?

**A:** MVP spełnia GDPR:
- **Nie zbieramy PII** (oprócz email użytkownika)
- **Nie trackujemy** behavior na audytowanych stronach
- **Data retention:** Raporty przez 6 miesięcy, potem usuwamy
- **User może usunąć** konto + wszystkie dane (GDPR Article 17)

Privacy Policy + Terms of Service w SETUP_GUIDE.md.

---

### Q: Jak secure są API keys?

**A:** Best practices:
- **Environment variables** (nie w kodzie)
- **Railway secrets** (encrypted at rest)
- **Rotation:** Co 90 dni (manual w MVP, auto w Etap 2)
- **Scope:** Minimum permissions (Claude API: tylko messages.create)
